# Project Compounding

Use this template when a project should get faster and more useful over time.

## Core Loop

The project should compound through this loop:

1. Context accumulation.
2. Asset accumulation.
3. Process automation.
4. Lower cost for the next task.

The goal is not only to complete the current task, but also to preserve reusable work that makes future tasks easier.

## Durable Assets To Consider

- Project instructions and rules.
- README files.
- Source indexes and reading maps.
- Concept inventories.
- Usage scenarios for common combinations of concepts.
- Review questions.
- Prompt templates.
- Scripts and repeatable workflows.
- Test cases or verification checklists.
- Decision records.
- Error logs that teach something durable.
- Archives of recurring outputs.

## When To Create An Asset

Create or propose a durable asset when:

- the same context is likely to be needed again;
- the user has a recurring preference;
- a task involves repeatable steps;
- a concept is difficult, central, or counterintuitive;
- future retrieval would otherwise require reconstructing the same work;
- an output can become a template, checklist, index, or archive.

Do not create a new file for every small exchange. Prefer lightweight updates to existing assets.

## Suggested Files

- `project_instruction.md`: project-specific instructions.
- `learning_assets.md`: mastery categories and review queue.
- `concepts.md`: key concepts, definitions, examples, and counterintuitive points.
- `usage_scenarios.md`: common situations where multiple concepts are used together.
- `review_questions.md`: active-recall questions grouped by topic.
- `learning_log.md`: what was studied, what was difficult, and what to revisit.
- `source_index.md`: important files, links, sections, page ranges, and how to use them.
- `decision_log.md`: important project decisions and why they were made.
- `templates/`: reusable prompts, workflows, or output formats.

## Review Habit

Periodically ask:

1. What did we learn that should be easier to retrieve next time?
2. What did the user struggle with that should enter the review queue?
3. What repeated work can become a template or script?
4. What source context should be indexed?
5. What instruction should be proposed for future projects?
